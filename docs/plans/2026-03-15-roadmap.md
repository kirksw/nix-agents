# nix-agents Roadmap

A self-optimizing personal AI operating system. One repo that controls how every AI coding tool behaves across your life — personal, work, per-project. It observes its own performance, tracks context across sessions, and improves over time. The tools are interchangeable; the system is yours.

## Architecture

```
+------------------------------------------------------+
|  Human                                                |
|  Preferences, ADHD context, communication style       |
|  Injected into ALL agents as baseline behavior        |
+------------------------------------------------------+
|  Profiles                                             |
|  Path-based default + per-project override            |
|  Controls: providers, keys, agents, skills, mcps,     |
|            hooks, model choices, permissions           |
+------------------------------------------------------+
|  Definitions                                          |
|  Agents, Skills, Hooks, MCP servers                   |
|  Tool-agnostic, composable                            |
+------------------------------------------------------+
|  Generators (best-effort per tool)                    |
|  opencode, claude, codex, cursor, amp, pi             |
+------------------------------------------------------+
|  Packages + Auto-update                               |
|  Nix derivations for tool binaries                    |
|  Updater that tracks upstream releases                |
+------------------------------------------------------+
|  agent-observe                                        |
|  Session tracking, cost, delegation, skill efficacy   |
|  Feeds back into optimization                         |
+------------------------------------------------------+
|  Runtimes (future)                                    |
|  Local (wrapped binary) | k8s (job/cron)              |
+------------------------------------------------------+
```

### 1.2 Human context module

A new module type (`human`) that defines communication preferences. Generators inject this as a preamble into all agent prompts.

**Module definition** (`modules/human.nix`):

```nix
{
  options.human = lib.mkOption {
    type = types.submodule {
      options = {
        name = lib.mkOption { type = types.str; };
        cognitiveStyle = lib.mkOption {
          type = types.nullOr (types.enum [
            "adhd"           # Needs pre-announced actions, structured updates, clear decision points
            "dyslexia"       # Needs simple language, short sentences, avoid dense text blocks
            "detail-focused" # Prefers exhaustive info, edge cases surfaced, minimal assumptions
            "high-level"     # Prefers summaries, abstractions, minimal implementation detail
            "visual"         # Prefers diagrams, tables, structured output over prose
          ]);
          default = null;
          description = "Cognitive style hint — agents adapt communication accordingly";
        };
        context = lib.mkOption {
          type = types.lines;
          default = "";
          description = "Free-form communication preferences, working patterns, domain expertise";
        };
        rules = lib.mkOption {
          type = types.listOf types.str;
          default = [];
          description = "Hard rules all agents must follow when interacting with this human";
        };
      };
    };
  };
}
```

**How cognitiveStyle shapes agent behavior**: The `cognitiveStyle` field is a shorthand that expands into a set of communication rules injected into agent prompts. Examples:

| Style | Agent behavior adaptations |
|-------|---------------------------|
| `adhd` | State intent before acting. Structured progress updates. Surface decisions clearly. Track session context for resumption. |
| `dyslexia` | Short sentences. Simple vocabulary. Avoid walls of text. Use bullet points and whitespace. |
| `detail-focused` | Surface edge cases proactively. Explain assumptions. Show full reasoning. List alternatives considered. |
| `high-level` | Lead with summary. Hide implementation detail unless asked. Emphasize outcomes over process. |
| `visual` | Prefer tables, diagrams, and structured output. Minimize prose. Use mermaid/ASCII where helpful. |

The `context` and `rules` fields allow further customization beyond the preset styles. All three are composable — a human can set a `cognitiveStyle` and add additional rules on top.

**Generator integration**: Each generator prepends the resolved human context (cognitive style rules + context + rules) to every agent's system prompt. The exact format depends on the tool (e.g., Claude Code puts it in CLAUDE.md, OpenCode in agent frontmatter).

### 1.3 Profile system

Profiles compose definitions + human context + provider config. Two resolution mechanisms:

1. **Path-based default**: directory prefix matching (e.g., `~/git/github.com/kirksw/` -> personal)
2. **Per-project override**: a local file in the project root selects a profile explicitly

**Module definition** (`modules/profile.nix`):

```nix
{
  options.profiles = lib.mkOption {
    type = types.attrsOf (types.submodule {
      options = {
        pathPrefixes = lib.mkOption {
          type = types.listOf types.str;
          description = "Directory prefixes that auto-select this profile";
        };
        providers = lib.mkOption {
          type = types.attrsOf providerType;
          description = "Model providers and credential sources";
        };
        agents = lib.mkOption {
          type = types.listOf types.str;
          description = "Which agents are active in this profile";
        };
        skills = lib.mkOption {
          type = types.listOf types.str;
          description = "Which skills are active in this profile";
        };
        mcpServers = lib.mkOption {
          type = types.listOf types.str;
          description = "Which MCP servers are active in this profile";
        };
        human = lib.mkOption {
          type = types.nullOr humanType;
          default = null;
          description = "Override human context for this profile";
        };
        tierMapping = lib.mkOption {
          type = types.nullOr (types.attrsOf types.str);
          default = null;
          description = "Override model tier mapping for this profile";
        };
        permissions = lib.mkOption {
          type = types.nullOr permissionsType;
          default = null;
          description = "Override default permissions for this profile";
        };
      };
    });
  };
}
```

**Example usage**:

```nix
{
  profiles.personal = {
    pathPrefixes = [ "~/git/github.com/kirksw" ];
    providers.anthropic = {
      credentialSource = "protonpass";  # or "apple-keychain"
      credentialRef = "Anthropic API Key";
    };
    agents = [ "code-monkey" "the-architect" "10xBEAST" "bottleneck" "chaos-demon" "code-red" "scribe" "explore" ];
    skills = [ "nix-coding-style" "nix-flake-ops" "nix-module-workflow" "secrets-management" "skill-creator" "swe-pruner-mcp" ];
  };

  profiles.work = {
    pathPrefixes = [ "~/git/github.com/lunarway" ];
    providers.anthropic = {
      credentialSource = "apple-keychain";
      credentialRef = "lunarway-anthropic";
    };
    # Maybe stricter permissions, different model tiers, fewer agents
    permissions.bash = "ask";
    tierMapping.fast = "anthropic/claude-haiku-4-5";
  };
}
```

**Profile resolution at runtime**: The wrapped binary (or a resolver script) checks `$PWD` against `pathPrefixes` to select a profile. If a `.nix-agents-profile` (or similar) file exists in the project root, it overrides path-based detection.

### 1.4 Provider module

Defines model providers with credential management.

**Credential sources** (in priority order):
1. **protonpass cli** (`protonpass-cli item get <ref>`)
2. **Apple Keychain** (`security find-generic-password`)
3. **Environment variable** (fallback)
4. **sops-nix** (for NixOS-managed secrets)

```nix
providerType = types.submodule {
  options = {
    credentialSource = lib.mkOption {
      type = types.enum [ "env" "protonpass" "apple-keychain" "sops" ];
      default = "env";
    };
    credentialRef = lib.mkOption {
      type = types.str;
      description = "Key name, env var name, or sops path depending on source";
    };
    envVar = lib.mkOption {
      type = types.str;
      description = "Environment variable name to set for the tool";
      example = "ANTHROPIC_API_KEY";
    };
  };
};
```

**Generator integration**: Wrapped binaries resolve credentials at launch time and set the appropriate env vars before exec-ing the tool.

### 1.5 Add cursor generator

Best-effort generator for Cursor's config surface.

**Output**: `.cursor/rules/*.mdc` files (one per agent/skill), `.cursor/mcp.json` for MCP servers.

Cursor rules use MDC format with frontmatter:
```
---
description: <agent description>
globs:
alwaysApply: true/false
---
<prompt content>
```

Limitations: Cursor has no native subagent delegation — agents become rules/context that guide behavior. Skills map directly to rules. MCP servers map to `.cursor/mcp.json`.

### 1.6 Add amp generator

Best-effort generator for Amp's config surface.

**Output**: `amp.yaml` configuration file, `AGENTS.md` for orchestration context.

Limitations: Amp is more batteries-included. Generator focuses on skills (as instructions) and MCP server config. Agent delegation may be limited to prompt-level guidance.

## Phase 2: Hooks + observability

Start collecting data about how agents perform.

### 2.1 Hook module

New module type. Hooks are shell scripts triggered on events. Generators map these to each tool's native hook mechanism where supported, or wrap the tool binary to inject them.

**Events**:
- `session-start` — agent session begins
- `session-end` — agent session ends
- `delegation` — one agent delegates to another
- `skill-invoked` — a skill is activated
- `human-decision` — user accepts/rejects/modifies agent output
- `commit` — a commit is created
- `error` — agent encounters an error

**Module definition**:

```nix
{
  options.hooks = lib.mkOption {
    type = types.attrsOf (types.submodule {
      options = {
        event = lib.mkOption { type = types.enum [ "session-start" "session-end" "delegation" "skill-invoked" "human-decision" "commit" "error" ]; };
        command = lib.mkOption { type = types.str; };
        package = lib.mkOption { type = types.nullOr types.package; default = null; };
      };
    });
  };
}
```

**Tool mapping**:
- Claude Code: native hooks system (`hooks` in settings.json)
- OpenCode: hook support TBD (may need wrapper-level injection)
- Others: wrapper-level pre/post scripts

### 2.2 Session context

Hooks write structured session state to a known location on session-end:
- What project/repo was being worked on
- What was accomplished (summary)
- What was left incomplete
- Active branch, last commit
- Duration, token usage

**Storage**: `~/.local/share/nix-agents/sessions/<profile>/<project>/`

A "resume" skill reads the latest session file and orients the agent at session-start. This enables "pick up where I left off" across sessions and tools.

A "status" query (via CLI or MCP) aggregates across all projects: "what projects do I have going on, where can I resume?"

### 2.3 agent-observe service

Standalone service in this repo under `services/agent-observe/`.

**Initial implementation**: lightweight HTTP service + SQLite storage.

**Data model**:
- Sessions: id, profile, project, tool, start_time, end_time, token_usage, cost
- Delegations: session_id, from_agent, to_agent, timestamp, duration
- Skills: session_id, skill_name, skill_version, timestamp
- Decisions: session_id, type (accept/reject/modify), timestamp, context

**Inputs**: hooks POST structured events to the service.

**Outputs**:
- CLI query interface (`nix run .#observe -- status`)
- MCP server so agents can query their own performance data
- Simple web dashboard (stretch goal)

### 2.4 Dashboard / query interface

- CLI: `nix run .#observe -- sessions --last-week`
- MCP server: agents can ask "what did I work on yesterday?" or "which skills are performing well?"
- Future: web UI

### Phase 2 exit criteria

- Every agent session produces structured telemetry
- Session state persists across sessions, enabling "resume where I left off"
- `nix run .#observe -- status` shows active projects and recent sessions
- Cost tracking works for at least claude and opencode

## Phase 3: Self-optimization

Use collected data to improve the system.

### 3.1 Skill versioning

Skills carry a `version` field. Hooks record which version was active during each session. This creates a history of skill performance over time.

```nix
{
  skills.nix-coding-style = {
    version = "1.2.0";
    # ...
  };
}
```

### 3.2 Efficacy metrics

agent-observe correlates:
- Skill version + model + outcome signals (accept/reject rate)
- Delegation patterns + completion rate
- Cost per successful outcome

Surface insights:
- "skill X v1.3 has 40% lower acceptance rate than v1.2 — consider reverting"
- "bottleneck on balanced model has same acceptance rate as powerful — save 60%"
- "chaos-demon is delegated to 2% of sessions — consider removing from default profile"

### 3.3 Prompt tuning suggestions

agent-observe identifies underperforming skills/agents and generates:
- Diff suggestions to skill prompts
- Model tier change recommendations
- Agent delegation graph optimizations

These can be surfaced as:
- CLI output from `nix run .#observe -- optimize`
- A PR-style workflow: generate changes, human reviews and accepts

### 3.4 Model cost/quality optimization

Given efficacy data, recommend tier changes per agent per profile. Work agents might prioritize speed; personal agents might prioritize quality.

### 3.5 Auto-update mechanism

A nix derivation updater that:
- Tracks upstream releases of all supported tools (codex, cursor, amp, claude, opencode, pi)
- Bumps version pins in `targets/*/package/versions.json`
- Runs `nix flake check` to validate
- Checks for new config surface area in updated tools and flags it for generator updates

Implementation: a nix app (`nix run .#update`) or a GitHub Action / scheduled k8s job.

### Phase 3 exit criteria

- Skills are versioned and efficacy is tracked
- `nix run .#observe -- optimize` produces actionable recommendations
- Auto-update keeps tool binaries current with minimal manual intervention

## Phase 4: Remote runtimes

Agents that run without you.

### 4.1 k8s job generator

- No decisions to surface (full autonomy within guardrails)

### 4.3 Cron/event-driven workflows

- Nightly research agent that summarizes developments in tracked repos
- Environment monitoring agent that checks infrastructure health

### 4.4 Result delivery

How remote agents report back:
- Create PRs with findings/changes
- Post to Slack/Teams

- Email digest

### Phase 4 exit criteria

- A research agent runs nightly in k8s and delivers findings as a PR or report
- Non-interactive profiles enforce safety guardrails
- agent-observe aggregates both local and remote session data

## Design principles

1. **Portability over purity** — best-effort support for each tool is better than perfect support for one
2. **Collect data, optimize later** — instrument first (Phase 2), analyze second (Phase 3)
3. **Human in the loop** — the system suggests improvements, the human decides
4. **Nix for reproducibility** — every config is a derivation, every tool version is pinned
5. **Open standards where they exist** — proprietary approaches only where the benefit is large
6. **Dog-food everything** — this repo itself uses the agents it defines
